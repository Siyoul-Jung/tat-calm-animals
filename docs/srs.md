## 1. 프로젝트 개요

### 1.1 배경

TATLife.com은 1993년 Tapas Fleming이 개발한 TAT®(Tapas Acupressure Technique)의 공식 플랫폼입니다. 현재 사이트는 WordPress + WooCommerce + ThriveSuite + Amelia + 10개 이상의 서드파티 플러그인으로 운영되고 있습니다.

재구축은 세 가지 핵심 이유로 추진됩니다:

**1. 전략적 포지셔닝 전환**

TAT®는 동물을 치유의 주된 진입점으로 내세우는 새로운 포지셔닝으로 전환합니다. "동물을 도우면, 당신 안에서도 무언가 일어난다"는 철학은 TAT® 수련의 본질을 반영합니다 — 동물과 인간의 치유는 분리될 수 없습니다. 현재 사이트는 이 비전을 담아내지 못합니다. 프로그램은 분리되어 있고, 동물-인간 연결이 홈페이지에서 보이지 않으며, 전체를 아우르는 명확한 내러티브가 없습니다.

또한 기존 프로그램 구조(TAT On-Demand, Healing ACEs Plus, TAT for Animals가 각각 별도 구독)를 단순한 2단계 멤버십 모델로 통합하여, 신규 방문자의 진입 장벽을 낮추고 장기 유지율을 높입니다.

**2. 사용자 경험 전면 개편**

현재 홈페이지는 정보 중심, 설명 우선입니다. 방문자는 TAT®가 무엇을 하는지 감각적으로 느끼기 전에 설명 텍스트 벽을 먼저 만납니다. 사이트 분석을 통해 확인된 주요 문제점:

- 정보 과부하 — 감정적 연결 없이 읽어야 할 내용이 너무 많음
- 신규 방문자를 위한 명확한 여정 없이 프로그램이 사이트 전반에 분산
- 커뮤니티 요소(주간 그룹 세션, 라이브 웨비나)가 깊이 묻혀 있음
- 사이트가 TAT®를 설명하는 데 그치고, 방문자가 경험하게 하지 않음
- 내비게이션이 복잡하고 하위 메뉴와 프로그램 카테고리가 겹침

새 사이트는 경험 우선입니다: 방문자는 설명 전에 먼저 느낍니다. 스크롤 기반 스토리텔링, 인터랙티브한 감정 고리(7가지 믿음), 미니멀한 디자인으로 기존의 정보 밀집 레이아웃을 대체합니다.

**3. 기술적 한계**

현재 WordPress 스택에는 10개 이상의 상호의존 플러그인이 쌓여 있어 지속적인 불안정성을 초래합니다. 사이트 관리자 Jez가 확인한 주요 문제점:

- Amelia 크론 스케줄러가 플러그인 업데이트 후 멈춤 — 매번 수동 재시작 및 테스트 필요
- 주간 그룹 TAT 상품을 매달 수동으로 생성 및 설정해야 함
- 구독 취소가 이메일로 수동 처리됨 — 셀프서비스 없음
- 인증 콘텐츠가 분산되어 감사 및 업데이트 어려움
- 플러그인 업데이트 후 연동이 깨지는 경우 발생 — 사용자 신고 후에야 발견
- 내비게이션 관련 고객 지원 이메일이 빈번함 — UX가 직관적이지 않음

### 1.2 프로젝트 목표

tatlife.com을 전면 재구축합니다 — 새로운 "TAT® Calm for Animals (and You)" 포지셔닝을 반영하고, 사이트 관리자의 운영을 자동화하며, 첫 방문부터 지속적인 참여까지 매끄러운 멤버 경험을 제공하는 더 빠르고 직관적인 플랫폼을 구현합니다.

### 2.1 전면 재구축을 선택한 이유 (Headless WordPress 대비)

현재 WordPress 스택에는 10개 이상의 상호의존 플러그인이 있습니다. 주요 문제점:

- Amelia 크론 스케줄러가 플러그인 업데이트마다 멈춤 — 매번 수동 재시작 필요
- 주간 그룹 TAT 상품을 매달 수동 생성해야 함
- 구독 취소가 이메일로 수동 처리됨 — 셀프서비스 없음
- 인증 콘텐츠가 분산되어 감사 어려움
- 고객 내비게이션 이메일 빈번 — UX 직관성 부족
- 플러그인 충돌로 업데이트 후 기능 이상 — 사용자 신고 후에야 발견

전면 재구축으로 플러그인 의존성을 완전히 제거하고, TAT®의 필요에 맞게 설계된 통합 자동화 시스템을 구현합니다.

## 5. 기능 요구사항

### 5.1 인증 및 사용자 역할

Supabase Auth를 통한 인증 (이메일/비밀번호 + 선택적 Google OAuth + Magic Link).

| 역할 | 접근 수준 |
|------|---------|
| `guest` | 공개 페이지만 |
| `subscriber` | 구독 단계별 콘텐츠 라이브러리 |
| `session_holder` | 세션 예약 + 크레딧 |
| `certification_student` | 학생 리소스 영역 |
| `certified_pro` | Pro 리소스 + Pro 할인 + CEU 추적 |
| `certified_trainer` | 트레이너 리소스 + 트레이너 할인 + CEU 추적 |
| `admin` | 전체 접근 + Sanity CMS 관리 |

### 5.2 멤버십 및 콘텐츠 접근

- Help Yourself 단계: Healing ACEs 라이브러리 + TAT for Animals 라이브러리 접근
- Professional Support 단계: 하위 단계 전체 + 라이브 웨비나 + 이전 녹화 접근
- 콘텐츠 접근은 Supabase RLS로 데이터베이스 수준에서 제어
- Stripe 웹훅을 통해 결제 성공 즉시 접근 부여
- 구독 취소 시 청구 기간 종료 후 접근 제거
- My Account 대시보드에서 접근 가능한 모든 콘텐츠를 한 화면에서 확인

### 5.3 세션 예약 (Cal.com)

- Stripe로 세션 크레딧 구매 → Cal.com 예약 링크 자동 발송
- Tapas Fleming과 30분 또는 60분 세션 예약
- 예약 확정 시 Zoom 링크 자동 생성 및 발송
- 72시간 취소 정책 — 72시간 이내 취소 시 환불 없음
- 세션 크레딧 구매일로부터 12개월간 유효
- 기존 패키지 코드 (2025년 6월 이전) 할인 쿠폰으로 지원
- 72시간 전까지 일정 변경 가능
- 자동 리마인더: 세션 24시간 전 + 1시간 전

### 5.4 이벤트 및 웨비나

- Sanity CMS로 이벤트 관리 — 수동 상품 생성 불필요
- 주간 그룹 TAT: 자동 반복 슬롯, 매달 수동 설정 불필요
- Stripe 결제 → Zoom 링크 자동 발송
- 구독자 웨비나 (ACEs, Animals): 구독 단계 기반 자동 초대

## 6. 자동화 요구사항 (Make.com)

모든 자동화 워크플로우는 Make.com으로 운영 — WordPress나 플러그인 인프라와 독립적.

### 6.1 결제 및 접근

- Stripe 결제 성공 → Supabase 역할 업데이트 → 환영 이메일 → 콘텐츠 접근
- 구독 취소 → 기간 종료 후 접근 제거 → 오프보딩 이메일
- 세션 구매 → Cal.com 예약 링크 자동 발송
- 이벤트 등록 → Zoom 링크 즉시 발송

### 6.2 리마인더 및 알림

- 세션: 24시간 전 + 1시간 전 Zoom 링크 포함 리마인더
- 웨비나: 24시간 전 + 1시간 전 리마인더
- 세션 크레딧 만료: 30일 전 경고 + 7일 전 경고
- 구독 갱신: 3일 전 사전 안내
- CEU 갱신: 자격증 만료 60일 전 경고

### 6.3 고객 지원 자동화

- 문의 양식 제출 시 카테고리별 담당 수신함으로 라우팅
- 구독 취소: My Account를 통한 셀프서비스
- 환불 요청: My Account에서 제출 → 관리자 알림 → 30일 이내 처리
- 내비게이션/지원 질문: AI 챗봇 FAQ 응대 *(Phase 2)*

## 7. 데이터베이스 스키마 (Supabase)

모든 테이블에 Row Level Security (RLS) 적용.

| 테이블 | 목적 |
|--------|------|
| `profiles` | 사용자 데이터 + 역할 (Supabase auth.users 확장) |
| `subscriptions` | Stripe ID 및 상태를 포함한 활성 구독 |
| `content_access` | 구독 기반 사용자-콘텐츠 라이브러리 매핑 |
| `session_credits` | 구매/사용/잔여 크레딧 및 만료일 |
| `session_bookings` | Zoom 링크 및 상태를 포함한 Cal.com 예약 |
| `events` | 접근 수준, CEU 학점, Zoom 정보를 포함한 전체 이벤트/웨비나 |
| `event_registrations` | 이벤트별 사용자 등록 및 참석 여부 |
| `ceu_records` | 사용자별/이벤트별/연도별 획득 CEU 학점 |
| `orders` | Stripe 결제 ID를 포함한 전체 구매 기록 |
| `order_items` | 주문별 품목 |
| `certifications` | 자격증 등록 및 완료 상태 |
| `certified_professionals` | 위치 정보를 포함한 Pro 및 트레이너 공개 디렉토리 |

## 9. 비기능 요구사항

### 9.1 성능
- Core Web Vitals: LCP < 2.5s, FID < 100ms, CLS < 0.1
- 모바일 및 데스크톱 Lighthouse 점수 > 90
- 빠른 글로벌 제공을 위한 CDN 비디오 스트리밍

### 9.2 보안
- 모든 사용자 데이터에 Supabase RLS 적용
- Stripe가 모든 결제 데이터 처리 — 카드 정보 저장 없음
- Cloudflare DDoS 보호 및 SSL
- 모든 API 키에 환경 변수 사용

### 9.3 접근성
- WCAG 2.1 AA 준수
- 키보드 내비게이션 지원
- 스크린 리더 호환
- 접근성 기준을 충족하는 색상 대비율

### 9.4 콘텐츠 관리 (Jez)
- Sanity CMS: 코드 없이 비디오, 이벤트, 후기 추가/편집
- Sanity 항목 생성 시 사이트 이벤트 캘린더 자동 업데이트
- PDF 이벤트 일정 관리 불필요
- Sanity를 통한 비디오 라이브러리 관리 + 자동 권한 설정

## 10. 마이그레이션 계획

### 10.1 마이그레이션 데이터

| 데이터 | 마이그레이션 방법 |
|--------|----------------|
| 활성 구독자 | WooCommerce에서 내보내기, Supabase로 가져오기, 비밀번호 재설정 안내 발송 |
| 주문 이력 | CSV 내보내기, Supabase orders 테이블로 가져오기 |
| 세션 크레딧 & 패키지 코드 | 기존 코드를 Supabase session_credits에 매핑 |
| 비디오 라이브러리 | Vimeo에서 Sanity CMS로 매핑 (Bunny Stream 마이그레이션은 선택적 Phase 2) |
| 블로그/콘텐츠 | WordPress XML 내보내기 → Sanity 가져오기 |
| 공인 전문가 디렉토리 | 현재 플러그인에서 Supabase certified_professionals로 내보내기 |

### 10.2 마이그레이션 단계

1. **Phase 1** — 기존 WordPress 사이트 전체 데이터베이스 백업
2. **Phase 2** — 스테이징 환경에서 새 사이트 구축 및 테스트
3. **Phase 3** — 스테이징에서 데이터 마이그레이션 및 검증
4. **Phase 4** — 병행 운영: 두 사이트 동시 운영 2~4주
5. **Phase 5** — 새 사이트로 DNS 전환
6. **Phase 6** — 모니터링 및 이슈 수정, WordPress 폐기
